# WatchTower — Natural Language Camera Intelligence

**Turn any camera into an AI agent with plain English.**

Domain: YHacks 2026 hackathon project (March 28-29, 2026)
Team: 4 members

## What This Is

A real-time camera monitoring system where users describe what to watch for in natural language. An LLM translates those descriptions into detection rules that run against a live YOLO + MediaPipe pipeline. When rules trigger, a vision LLM narrates the scene in context.

## Tech Stack

- **Frontend:** Next.js 15 + React 19 + Tailwind v4 + shadcn/ui
- **Backend:** Python (FastAPI) + WebSockets
- **Computer Vision:** YOLO v8 (ultralytics) + MediaPipe Pose
- **LLM:** Claude / GPT-4o for rule parsing + alert narration
- **Real-time:** WebSocket streaming between backend and frontend

## Core Concepts

### Zones
Regions of interest on the camera frame. Can be auto-generated by a vision LLM analyzing a still frame, or manually drawn/edited by the user. Each zone has a name, coordinates (x, y, width, height as percentages), and color.

### Rules
Detection conditions expressed in natural language, compiled by an LLM into structured JSON. Each rule has:
- `name`: human-readable label
- `conditions`: array of checks (object presence, zone membership, pose state, count, duration, time window)
- `severity`: low / medium / high / critical
- `enabled`: on/off toggle

### Detection Primitives (What Rules Can Check)
| Primitive | Source | Description |
|-----------|--------|-------------|
| object_present | YOLO | Specific object class detected in frame |
| object_in_zone | YOLO + zones | Object bounding box overlaps a named zone |
| object_absent | YOLO | Expected object NOT detected |
| person_size | YOLO | Bounding box height ratio (child vs adult proxy) |
| person_pose | MediaPipe | Standing, sitting, lying, crouching, arms raised |
| person_falling | MediaPipe | Rapid downward velocity of hip/shoulder keypoints |
| count | YOLO | Number of objects of a given class |
| duration | Rule engine | Condition true for N consecutive seconds |
| time_window | Rule engine | Rule only active during specified hours |
| movement_speed | YOLO | Bounding box displacement between frames |
| stillness | YOLO | No significant movement for N seconds |

### Alerts
When a rule's conditions are met, an alert fires:
1. Captured frame is saved to the replay buffer
2. Frame is sent to a vision LLM for contextual narration
3. Alert appears in the log with narration, timestamp, severity, and captured image
4. In-app notification (sound + visual). Push notifications planned for future.

### Auto-Zone Generation
On camera connect, the system captures a still frame and sends it to a vision LLM with the prompt: "Identify areas of interest in this room for safety monitoring (doors, stairs, windows, counters, etc.). Return bounding boxes with labels." User can then edit/rename/delete/add zones.

### Replay
A circular buffer stores the last 30 minutes of frames (at reduced resolution/framerate to manage memory). A timeline scrubber shows alert markers. Clicking an alert jumps to that moment in the replay.

## Data Models

### Zone
```typescript
interface Zone {
  id: string
  name: string
  x: number      // percentage 0-100
  y: number      // percentage 0-100
  width: number   // percentage 0-100
  height: number  // percentage 0-100
  color: string
}
```

### Rule
```typescript
interface Rule {
  id: string
  name: string
  naturalLanguage: string  // original user input
  conditions: Condition[]
  severity: "low" | "medium" | "high" | "critical"
  enabled: boolean
  createdAt: string
}

interface Condition {
  type: "object_present" | "object_in_zone" | "object_absent" | "person_size"
       | "person_pose" | "person_falling" | "count" | "duration"
       | "time_window" | "movement_speed" | "stillness"
  params: Record<string, unknown>
}
```

### Alert
```typescript
interface Alert {
  id: string
  ruleId: string
  ruleName: string
  severity: "low" | "medium" | "high" | "critical"
  timestamp: string
  frameUrl: string        // base64 or blob URL of captured frame
  narration: string       // vision LLM description
  detections: Detection[] // what was detected when alert fired
}
```

### Detection
```typescript
interface Detection {
  class: string           // YOLO class name
  confidence: number
  bbox: { x: number; y: number; width: number; height: number }
  pose?: PoseKeypoints    // if person, MediaPipe keypoints
}
```

## LLM Prompts

### Rule Parsing System Prompt
```
You are a camera monitoring rule compiler. The user describes a monitoring rule in plain English. You must translate it into a structured JSON rule using ONLY the available detection primitives.

Available primitives: [list from table above]
Available zones: [dynamically injected from current zone list]

If the user's request cannot be fully captured by the available primitives, do your best approximation and explain what you approximated in the rule name.

Respond with ONLY valid JSON matching the Rule schema.
```

### Zone Generation System Prompt
```
You are analyzing a camera frame from inside a room/space. Identify areas that would be relevant for safety monitoring. For each area, provide:
- name: descriptive label (e.g., "Staircase", "Front Door", "Kitchen Counter")
- bounding box: x, y, width, height as percentages of frame dimensions

Return JSON array of zones. Be conservative — only identify clearly visible areas of interest.
```

### Alert Narration System Prompt
```
You are a home safety monitoring assistant. An alert has been triggered. Describe what you see in this camera frame in 1-2 sentences. Be specific about:
- Who/what is in the frame
- What they are doing
- Why this might be concerning
- Whether immediate action seems needed

Be calm and factual, not alarmist.
```

## Conventions

- No database — all state in memory (rules, zones, alerts, replay buffer)
- No authentication — runs locally
- No deployment — localhost only for the hackathon demo
- WebSocket is the only communication channel between frontend and backend
- All coordinates are percentages (0-100) relative to frame dimensions for resolution independence
- YOLO v8n (nano) model for speed — accuracy is sufficient for demo
- Frames sent to frontend as base64 JPEG via WebSocket
- LLM calls are async and non-blocking — detection pipeline never waits on LLM

## UI Rules (v1)

These rules govern all frontend code and design decisions. They can change at any time.

### Design Direction
Modern command center with tactical HUD elements on the camera feed. Think Bloomberg Terminal meets security operations center, not a consumer app.

### Theme
- Dark mode is the default and primary design target
- Light mode must also work (in-page toggle)
- Dark background: slate-950
- Light background: white
- No "in between" grays for backgrounds. Commit to dark or light.

### Colors
- Primary accent: cyan-400 (active detections, zone borders, interactive elements)
- Severity levels — use consistently everywhere:
  - Low: green-400
  - Medium: amber-400
  - High: orange-500
  - Critical: red-500
- Zone overlays: semi-transparent fills (10-20% opacity) with colored borders
- Bounding boxes: cyan with corner brackets, NOT full rectangles

### Typography
- Camera overlays and data readouts: JetBrains Mono (monospace)
- Dashboard UI (labels, buttons, headings): Inter or Geist Sans
- No decorative fonts
- No italic for emphasis. Use weight or color.

### Layout
- Single page dashboard. No page navigation.
- Dense information layout. Minimize whitespace. Data should feel packed but organized.
- Sharp corners on cards and containers. No rounded-xl or pill shapes.
- Camera feed is the largest element and visual anchor
- Panels around the feed: rules, alerts, zones, chat input

### Camera Feed Overlay Style
- Bounding boxes use corner brackets (L-shaped corners), not full rectangles
- Detection labels in monospace, small, positioned above the bounding box
- Zone outlines with subtle glow/bloom effect
- Thin grid or crosshair overlay (very subtle, low opacity)
- Timestamp and FPS counter in corner, monospace

### Icons
- Use Lucide icons only (ships with shadcn/ui)
- No emojis anywhere in the UI. Ever.
- Icons should be functional, not decorative

### Text Content
- No em dashes in UI text
- No marketing-style explainer text or taglines in the app UI
- No false claims about capabilities the system doesn't have
- Labels should be short, factual, and technical
- Alert narrations come from the LLM only, never hardcoded
- Error messages should be specific and actionable

### Animations
- Alerts: subtle flash or pulse on arrival, not bouncing or sliding
- Zone drawing: immediate visual feedback, no transition delays
- Theme toggle: instant swap, no crossfade
- Bounding boxes: update positions every frame with no interpolation (raw tracking feel)
- No loading spinners. Use skeleton placeholders or status text.

### Components
- shadcn/ui as the base component library
- Customize shadcn defaults to match the command center aesthetic (sharp corners, denser padding)
- No component should look like default unstyled shadcn

### What to Avoid
- Rounded corners larger than rounded-sm
- Gradient backgrounds
- Drop shadows (use borders or subtle glow instead)
- Placeholder stock images
- Lorem ipsum or dummy text
- "Powered by" or "Built with" badges in the UI
- Any text that makes it look auto-generated or vibe-coded
